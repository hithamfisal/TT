# Follow-Up Sheets Dashboard Deployment Guide

**Author:** Manus AI  
**Project:** Follow-Up Sheets Dashboard  
**Application type:** Static React dashboard with in-browser Excel workbook upload and local saved sessions

## 1. Deployment overview

The Follow-Up Sheets Dashboard is a **frontend-only React application**. It reads the uploaded Excel workbook directly in the browser, calculates dashboard metrics on the client side, and stores saved dashboard sessions in the user’s browser storage. Because the workbook parsing happens locally in the browser, no backend server or database is required for the current version.

| Area | Current behavior |
|---|---|
| Workbook upload | The user uploads an `.xlsx` workbook through the dashboard interface. |
| Data processing | Ticket rows are parsed and analyzed in the browser. |
| Saved sessions | Dashboard data is saved locally in the browser, so the user does not need to re-upload the workbook every time. |
| Deployment model | Static web application generated by Vite. |
| Backend requirement | Not required for the current dashboard. |
| Best hosting option | Manus built-in hosting is recommended because the project was created and checkpointed inside Manus. |

> **Recommended path:** Use Manus built-in publishing from the project interface. This keeps the deployment aligned with the current project checkpoint and avoids external hosting compatibility issues.

## 3. Local setup

To run the dashboard locally, first extract the ZIP archive, then open a terminal in the extracted project folder. The project uses Node.js and pnpm. Node.js is the JavaScript runtime used to execute development tooling, while pnpm is the package manager used by this project.[1] [2]

| Step | Command | Expected result |
|---|---|---|
| Enter the project folder | `cd follow-up-sheets-dashboard-web` | The terminal is inside the project root. |
| Install dependencies | `pnpm install` | The `node_modules` folder is recreated locally. |
| Start development server | `pnpm dev` | A local Vite development server starts. |
| Open the app | Use the local URL shown in the terminal, usually `http://localhost:3000` or another available port. | The dashboard opens in the browser. |

If `pnpm` is not available on your machine, install it using the official pnpm installation method for your operating system.[2]

## 6. Optional external hosting

Manus built-in hosting is recommended for this dashboard, especially because it was developed and checkpointed inside a Manus web project. If you still want to host it externally, the dashboard should be treated as a static Vite application. External hosting may work, but there can be compatibility differences around routing, analytics environment variables, and platform-specific build settings.

| Provider type | Typical build command | Typical output folder | Compatibility note |
|---|---|---|---|
| Static hosting platform | `pnpm build` | `dist/public` | Configure the platform to publish the generated static output folder. |
| Vercel-style hosting | `pnpm build` | `dist/public` | Ensure the output directory is set correctly, because this project’s production build places frontend files under `dist/public`. |
| Netlify-style hosting | `pnpm build` | `dist/public` | Add a redirect rule if client-side routing is expanded beyond the home page. |
| Traditional web server | `pnpm build` | `dist/public` | Copy the contents of `dist/public` to the server’s public web directory. |

If using external hosting, confirm these settings before deployment.

| Setting | Recommended value |
|---|---|
| Install command | `pnpm install` |
| Build command | `pnpm build` |
| Publish/output directory | `dist/public` |
| Node version | Use a current LTS version of Node.js unless your hosting provider requires another supported runtime. |

## 7. Workbook and saved-session behavior after deployment

The deployed dashboard does not include your uploaded workbook inside the application bundle. Each user uploads a workbook through the browser. After upload, the dashboard can save a local browser session, allowing that same browser to reopen the dashboard without uploading again.

| Scenario | Expected behavior |
|---|---|
| Same browser, same device | Saved dashboard session can be restored automatically or through the restore flow. |
| Different browser or device | The workbook must be uploaded again. |
| Browser storage cleared | Saved session is removed and the workbook must be uploaded again. |
| Updated workbook data | Upload the newer workbook and save a new local session. |

This behavior is intentional because the dashboard currently stores sessions locally in the browser rather than in a shared database.

## 8. Security and privacy notes

The dashboard processes workbook data in the browser. This reduces the need to send workbook contents to a backend server in the current version. However, users should still treat uploaded ticket data as operationally sensitive and publish the dashboard only with suitable visibility settings.

| Concern | Recommendation |
|---|---|
| Sensitive ticket data | Use appropriate Manus visibility settings before publishing. |
| Shared workstations | Clear the saved local session after use. |
| Multi-user access | Consider adding authentication and server-side storage in a future full-stack version. |
| Centralized saved sessions | Upgrade to a backend/database architecture if sessions must follow users across devices. |

## 10. Troubleshooting

| Issue | Likely cause | Suggested fix |
|---|---|---|
| `pnpm` command not found | pnpm is not installed globally. | Install pnpm, then rerun `pnpm install`. |
| Build fails after modifying code | TypeScript or JSX error. | Run `pnpm check`, fix the reported file and line, then rerun `pnpm build`. |
| Blank page after external deployment | Wrong output folder or routing configuration. | Confirm the publish directory is `dist/public`. |
| Saved session does not appear | Browser storage was cleared or a different browser/device is being used. | Upload the workbook again and save a new session. |
| Workbook uploads but metrics look wrong | Workbook headers or sheet structure changed. | Confirm the uploaded workbook still uses the same ticket register structure as the original Follow-Up Sheets workbook. |

## 11. Recommended next improvements

The current deployment is suitable for a static, browser-based dashboard. If you want the dashboard to become a shared operational tool, the most useful next improvement would be user authentication plus centralized session storage, so authorized users can access the same saved dashboard data from different devices. A second practical improvement would be a dedicated PDF-style executive report export that mirrors the workbook Dashboard sheet more closely. A third improvement would be chart drill-down pages, allowing users to click a severity, region, site, or status chart segment and inspect the underlying TT records immediately.
